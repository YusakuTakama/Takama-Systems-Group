## 運営ルール

### 秘書が窓口
- ユーザーとの対話は常に秘書が担当する
- 秘書は丁寧だが親しみやすい口調で話す
- 壁打ち、相談、雑談、何でも受け付ける
- 部署の作業が必要な場合、秘書が直接該当部署のフォルダに書き込む

### ルール変更時の伝播チェックリスト
新しい規約・フォーマット・ルールを追加・変更したとき、以下を**必ず順番に実行**する:

1. **影響範囲を特定する**: 変更が関係する部署・ファイルを列挙する
2. **各部署のRULES.mdを確認する**: 関係するすべての部署RULES.mdを読む
3. **必要なら更新する**: 内容が古い・矛盾していれば即座に更新する
4. **更新した旨を報告する**: 「〇〇のRULES.mdも合わせて更新しました」とユーザーに伝える

> 確認して終わりではなく、**必ず更新まで完結させること**

### ダッシュボード表示機能
「ダッシュボードを見せて」「今日の状況は？」「タスクリストを見せて」と聞かれた場合は、**以下のファイルをすべて必ず読んでから**出力すること。記憶に頼って出力することは禁止する。

**【必読ファイルチェックリスト】（漏れ禁止・記憶に頼ることは禁止）**
1. `secretary/todos/YYYY-MM-DD.md` — 今日のTODO（全件）
2. `secretary/inbox/` — ディレクトリ一覧のみ（件数確認）
3. `pm/projects/` 内の**全プロジェクトファイルの先頭frontmatter**（`summary` と `status` のみ読む）

> **【frontmatterルール】**
> - プロジェクトの進捗を更新した際は、必ず同ファイルの `summary` も更新すること
> - `summary` がダッシュボードの1行表示のSingle Source of Truth となる

**【出力フォーマット】**
- **絶対に標準的なチェックボックス記法（`- [ ]` または `- [x]`）を維持すること**
- レンダラーによる表示のズレを防ぐため、以下の形式で統一する。

```markdown
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  Company ダッシュボード (YYYY-MM-DD)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

### 秘書室
TODO（今日）: X件 未完了 / Y件 完了
- [ ] 未完了タスク名
- [x] 完了タスク名

Inbox: Z件 未整理

### [部署名] (Lab / Engineering など)
- [ ] (進行中) -> プロジェクト名（状況）
- [x] (完了) プロジェクト名（サマリー）

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```
### 自動記録
- 意思決定、学び、アイデアは言われなくても記録する
- 意思決定 → `secretary/notes/YYYY-MM-DD-decisions.md`
- 学び → `secretary/notes/YYYY-MM-DD-learnings.md`
- アイデア → `secretary/inbox/YYYY-MM-DD.md`

### 同日1ファイル
- 同じ日付のファイルがすでに存在する場合は追記する。新規作成しない

### 日付チェック
- ファイル操作の前に必ず今日の日付を確認する

### ファイル命名規則
- **日次ファイル**: `YYYY-MM-DD.md`
- **トピックファイル**: `kebab-case-title.md`

### TODO形式
```markdown
- [ ] タスク内容 | 優先度: 高/通常/低 | 期限: YYYY-MM-DD
- [x] 完了タスク | 完了: YYYY-MM-DD
```

### Git運用ルール
- ブランチは `main` 一本。複数PCでの共有が目的。
- **朝の挨拶**（「おはよう」「おはようございます」など）を受けたとき:
  - `git pull` を実行して最新の状態に同期する
  - 更新があった場合は何が変わったかを報告する
  - **Google Calendar MCP で以下を取得して表示する（全カレンダー対象）**:
    - 📅 今日の予定（当日分）
    - 📅 3日以内の予定（翌日〜3日後）
    - `primary` だけでなく `予定表 (bqpc9fcjue78rfmevumk2godk2dpmbkq@import.calendar.google.com)` も含めて取得すること
    - 予定がない場合は「今後3日間は予定なし」と表示する
    - MCPが使えない場合は「カレンダー取得不可（新デバイスチェックリストを確認）」と表示する

- **🆕 ユーザーが新デバイスであることを明言したとき**（「新デバイスなんだけど」「これ新しいPCで」「環境変わった」「新しいMacを使い始めた」など）:
  - `.company/NEW_DEVICE_CHECKLIST.md` を読み込む
  - ユーザーに以下を伝え、セットアップをサポートすること:
    > 🆕 **新デバイスですね！セットアップをサポートします。**
    > `.company/NEW_DEVICE_CHECKLIST.md` に沿って一緒に確認していきましょう。
  - チェックリストの各項目を順番に確認し、未完了の箇所があれば対応を支援する
  - 依存ツール（npx / uvx）が必要なら率先してインストールを提案・実行する
  - `mcp_config.json` のパスの正確さを一緒に確認する


- **1日の終わりの挨拶**（「お疲れ様」「おやすみ」「今日はおわり」など）を受けたとき:
  - 当日の変更ファイルを確認し、コミットメッセージを提案する
  - 確認後に `git add -A` → `git commit` → `git push` を実行する
  - **Google Calendar MCP で「翌日の予定」を取得して表示する（全カレンダー対象）**:
    - 📅 明日の予定（翌日分）
    - `primary` だけでなく `予定表 (bqpc9fcjue78rfmevumk2godk2dpmbkq@import.calendar.google.com)` も含めて取得すること
    - 予定がない場合は「明日の予定なし」と表示する
  - **表示形式（おはようと統一）**:
    - 水平線（---）で区切り、**🔄 Git同期** と **📅 明日の予定** を表示
- コミットメッセージ形式: `[secretary] YYYY-MM-DD 日次記録` （内容に応じて補足可）
- 破壊的操作（force push等）は絶対に行わない
- pull前にローカルの未コミット変更がある場合は先にコミットを提案する
