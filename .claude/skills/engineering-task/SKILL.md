# Engineering Task Skill（エンジニアリング・タスクスキル）

個人開発プロジェクトへのコード改修・機能追加・UI調整をプロ品質で管理するエンドツーエンドのワークフロー。

---

## Phase 1: 要件定義と承認

ユーザーの依頼を受けたら、まず実装プランを作成し、**合意を得ることを最優先**にする。

1. **現状調査**: 対象ファイルを `Read` で読み込み、現在の実装を把握する
2. **実装プランの提示**: 以下を含むプランをチャットで提示する
   - 実装のゴールと期待される挙動
   - 変更対象ファイルと具体的な変更内容
   - 懸念点やトレードオフ（あれば）
3. **フィードバックとブラッシュアップ**: 質問・指摘に回答し、プランを洗練させる
4. **実装移行のトリガー**: ユーザーが「よし、いこう」「実装開始して」など**明示的に許可したときのみ** Phase 2 へ進む。勝手に実装を開始してはならない

> **Why**: 認識の相違をコードレベルで検証する前に解消することで、手戻りをゼロにする。

---

## Phase 2: 実装

ユーザーの許可を得た後、プランに基づいて実装する。

- **1ファイルずつ、依存順に変更する**
- `Edit` ツールで的確なターゲット行を指定して編集する
- 変更後は `Read` で実際のコードを再確認し、意図通りに適用されたか確認する

**開発サーバーの確認（必要に応じて）:**

```bash
cd "<project_root>" && npm run dev
```

TypeScriptエラーがないか `Bash` でターミナルログを確認する。

---

## Phase 3: ユーザー確認と修正

実装完了後、**まずユーザーに結果を報告し、動作確認を依頼する。**

1. 実装完了をユーザーに伝え、確認を促す
2. 「ここが動かない」「ここをもう少し直して」などのフィードバックを受ける
3. 指摘がある場合は Phase 2 に戻り修正する
4. **「OK」「これで大丈夫」「完璧」など、明示的な承認を得るまで Phase 4 以降へ進まない**

> 一度の実装で完璧に仕上がることは稀という前提で、対話を通じてクオリティを磨き上げるフェーズ。

---

## Phase 4: ブラウザ検証（原則スキップ）

**このフェーズは原則スキップ。** ユーザーから「ブラウザで確認して」「動作させてみて」と明示的な指示があった場合のみ、`Agent` (Playwright) を起動する。

実行する場合のチェックリスト:

```
[ ] 実装した機能が期待通りに動作するか
[ ] スタイルが他の要素と視覚的に調和しているか
[ ] スマートフォンサイズ（375x667）で表示が崩れていないか
[ ] エッジケース（リストの端、空の状態）で問題がないか
```

---

## Phase 5: ドキュメント更新

**ユーザーの承認（Phase 3）を得た後**、以下を `Edit` で更新する:

| ファイル | 更新内容 |
|---|---|
| `spec.md` | 新機能の仕様・変更点を正確に反映 |
| `.company/engineering/docs/<project>.md` | 実装内容の技術概要を追記 |
| `.company/pm/projects/<project>.md` | 該当タスクを `[x]` にマーク |
| `.company/secretary/notes/YYYY-MM-DD-decisions.md` | 使用した技術・設計の意図を記録 |
| `.company/secretary/notes/YYYY-MM-DD-learnings.md` | 実装中に得た知見・ハマりどころを記録 |

---

## Phase 6: Git Push

ユーザーから「push して」という指示を受けた場合のみ実行する:

```bash
git -C "<absolute_project_root>" add .
git -C "<absolute_project_root>" commit -m "<type>: <summary of changes>"
git -C "<absolute_project_root>" push
```

> ワークスペース外（OneDrive等）で絶対パスが拒否される場合は相対パス（`../〜`）で迂回する。
