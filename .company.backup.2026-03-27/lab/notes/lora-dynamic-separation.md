> 初回記録: 2026-03-22

## 研究テーマ

**複数LoRAの動的分離における「ほぼ直交（0.9999）」の理論的解明とマルチタスク学習等への応用可能性**

### 目的
- 事前学習済みモデルに複数のLoRAを追加し、損失関数で各LoRAの部分空間を動的に直交化（分離）させる
- **主目的**: タスク間の干渉を防ぎ、継続学習（Continual Learning）の破滅的忘却（Catastrophic Forgetting）を防ぐ
- **副目的**: マルチタスク学習などでの効率的な知識分離

---

## 分離指標の定義

| 指標 | 略称 | 内容 |
|------|------|------|
| 直交 | orth | シンプルな直交性 |
| コーダル距離 | cd | Chordal Distance |
| 加重コーダル距離 | wcd | 特異値で重み付けしたコーダル距離（重要成分のみ分離） |

---

## 実験結果サマリー

### 実験1: 従来手法（単一Optimizer + λブースト）

- **手法**: `L_ce`（タスク損失）+ `L_orth`（直交化損失）を単一Adamで更新。λのハイパーパラメータ探索（100, 1000, 10000）
- **結果**:
  - λ=100, 1000 → 直交性指標が1.0に届かず、既存手法（InfLoRA）に届かない
  - **λ=10,000 → 指標が完全に1.0（完全直交）達成、InfLoRA並みの性能を達成**
  - ただしλ=10,000では制約が強すぎてL_ceが下がりきらない（初期タスク精度が低下）

### 実験2: Dual Optimizerの導入（メイン実験）

- **手法**: 各ミニバッチ内で「L_ceによる更新」と「L_orthによる更新」を別々のAdam（η_ce, η_orth）で交互に実行
- **結果**:
  - L_ceはλブースト時より適切に低下 ✅
  - **直交性指標が0.9999（ほぼ直交）に留まり、1.0に届かない** ❌
  - **このモデルを継続学習で評価 → 破滅的忘却を防げず、λブースト手法より大幅に性能低下** ❌

### 実験3: Dual Optimizer アブレーション + cd/wcd比較

- **手法**: η_ce=0.0005固定、η_orthを×2〜×100にして制約の強さを検証
- **結果**:
  - η_orthを極端に大きくすると、オーバーシュートにより逆に制約が大幅に弱まる
  - スイートスポット設定でも、タスク数増加につれて距離が1.0に届かなくなる
  - **wcd（加重コーダル距離）もタスクが進むとcdと同様に0.9999付近で妥協**
  - **継続学習の最終性能でcd vs wcdで差が出ない**（wcdの「重要成分のみ分離」の優位性が見えず）

## 2026-03-25 MTG ディスカッションメモ

### 低次元での数値検証結果
- 4次元空間内の2次元部分空間2つで検証。シナリオ1（メジャー離れ）/ シナリオ2（メジャー近接・マイナー離れ）の2パターン
- PCS・加重コーダル距離ともに期待通りの挙動。min-max正規化すれば同等に見える
- 教授評価: ミニマム実験としてはOK

### ヒートマップ可視化の提案（教授）
- 2個のLoRA（A, B）のSVDベクトルを重要度順に並べ、ペアごとの角度をヒートマップで表示する
- 対角（同順位ペア）が赤（離れている）、非対角が青（近い）なら期待通り
- **IoUの空間版イメージ**: スカラに潰さず空間ごとの関係を保存する指標
- マルチドメイン（3個以上）の場合は全組み合わせをクロスでプロットすればよい
- 加重コーダル距離との対応: 考え方は繋がるが式としては直接繋がらない → 別の観点として併用が妥当

### 高次元での課題
- 128次元等になると、メジャー・マイナーが10〜20個存在し、スカラ指標1個ではどの成分が効いているか不明
- 加重コーダル距離は「和をとる」ことで個別成分の情報を失う → 「和をなくしてバラバラにすれば意味のあるベクトルが得られるのでは」（教授）

### 「0.9999」問題の教授評価
- 干渉蓄積・臨界閾値の論文根拠について「一つの解釈としてはありそう」
- ただし、数値計算上の1.0にも10^-16程度の誤差がある → 論文の議論はそこを区別していないとの指摘
- 臨界閾値に達するステップ数が直交精度で異なるという解釈が妥当か → 引き続き精査必要
- **思考実験**: タスク切り替え時に射影すれば累積誤差は解消されるが性能は下がる。InfLoRA等は最初から射影空間内で学習するためその「ガクン」がない

### 今後の研究方向
- 「損失1にする手法」と「1でなくても意味がある応用」の両輪で進める
- 継続学習でのヒートマップ時系列変化の観察は非常に有意義（教授）
- AIエージェントのリサーチ活用を検討（CLIだけでなくGUI操作も）

## チェックポイントの構造（2026-03-26判明）

### 保存形式の理解

**重要な発見**:
- `task_19.pth` という**1つのチェックポイント**に、**task 0〜19の全20個のLoRA**が含まれている
- 各タスクのLoRAパラメータは**ModuleList**として並列に保存されている

### InfLoRAb5の実装（vit_inflorab5.py）

```python
class Attention_LoRA(nn.Module):
    def __init__(self, dim, num_heads=8, ..., r=64, n_tasks=10):
        ...
        # 各タスクごとにLoRAのペアを保持
        self.lora_A_k = nn.ModuleList([nn.Linear(dim, r, bias=False) for _ in range(n_tasks)])
        self.lora_B_k = nn.ModuleList([nn.Linear(r, dim, bias=False) for _ in range(n_tasks)])
        self.lora_A_v = nn.ModuleList([nn.Linear(dim, r, bias=False) for _ in range(n_tasks)])
        self.lora_B_v = nn.ModuleList([nn.Linear(r, dim, bias=False) for _ in range(n_tasks)])
```

### チェックポイント内のキー構造（推定）

```
state_dict = {
    "blocks.0.attn.lora_A_k.0.weight": ...,  # Task 0のlora_A_k
    "blocks.0.attn.lora_B_k.0.weight": ...,  # Task 0のlora_B_k
    "blocks.0.attn.lora_A_v.0.weight": ...,  # Task 0のlora_A_v
    "blocks.0.attn.lora_B_v.0.weight": ...,  # Task 0のlora_B_v

    "blocks.0.attn.lora_A_k.1.weight": ...,  # Task 1のlora_A_k
    "blocks.0.attn.lora_B_k.1.weight": ...,  # Task 1のlora_B_k
    ...

    "blocks.0.attn.lora_A_k.19.weight": ..., # Task 19のlora_A_k
    "blocks.0.attn.lora_B_k.19.weight": ..., # Task 19のlora_B_k
    ...
}
```

### ヒートマップ可視化への影響

**正しい実装方針**:
1. **1つのチェックポイント**（例: `task_19.pth`）を読み込む
2. 同じ層（例: `blocks.0.attn`）から**異なるタスクID**のlora_B_kを抽出
   - Task 0: `state_dict["blocks.0.attn.lora_B_k.0.weight"]`
   - Task 1: `state_dict["blocks.0.attn.lora_B_k.1.weight"]`
3. これら2つのLoRA_B行列の列空間の基底ベクトル間の角度をヒートマップ化

**間違った理解（修正済み）**:
- ~~異なるチェックポイント（`task_0.pth` vs `task_1.pth`）を比較する~~ ❌
- 正しくは同じチェックポイント内の異なるタスクを比較 ✅
